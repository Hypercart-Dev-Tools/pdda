## How this plan was shaped

**The Iron Triangle choice is the primary scoping lever.** It does not just label
this plan — it decides **how many milestones exist, how fast they move, and how
many validation gates gate them.** Before generating milestones, load the
matching branch doc and follow its milestone-shape guidance:

- **Faster** → `iron-triangle-faster.md` (fewest milestones, minimal gates,
  ship-and-learn)
- **Better** → `iron-triangle-better.md` (more, smaller milestones; a gate per
  milestone)
- **Cheaper** → `iron-triangle-cheaper.md` (Fast Track option + **UX/Dev Ratio
  Discipline** — messaging/UX are first-class deliverables, not a final phase)

The branch doc sets the **shape** (count, pace, gate density, "good enough"
bar). This template sets the **format** of each individual milestone. Time
horizon from Q9 calibrates granularity within that shape.

**Do not emit the default 5-milestone ladder regardless of the choice.** A
"Faster / Cheaper" build should have fewer, thinner milestones than a "Better"
build. Read the branch doc and size the plan to it.

**Universal rules for milestone scope (all branches):**
- Each milestone is a **vertical slice** that delivers independently testable
  value.
- Each milestone has explicit, **verifiable** completion criteria.
- Milestones depend only on previously completed milestones.
- No milestone adds scope beyond what's locked in the PRD.
- **Table stakes** (modern design + mobile-responsive) apply from the first
  user-facing milestone — never deferred to a "polish" phase.

## Reference milestone ladder (adapt to the branch, do not copy verbatim)

A conventional foundation-first ordering, to be **compressed or expanded** by the
Iron Triangle branch:

1. **Foundation** — schema (PRD §5), auth, project scaffolding, CI/CD skeleton.
2. **Core data layer** — CRUD endpoints, validation, authorization middleware.
3. **Primary user flows** — the P0 screens and end-to-end flows (PRD §4), with
   table-stakes responsive design applied here, not later.
4. **Edge cases, polish & non-functional** — empty states, error boundaries,
   rate limiting, performance (N+1), accessibility pass.
5. **Advanced features & integrations** — P1/P2 requirements (PRD §3.2/§3.3),
   third-party integrations.

- **Faster / Cheaper** typically collapse this to ~2–3 milestones and defer or
  drop 4–5 until signal appears.
- **Better** typically keeps all five and may split them further, each with its
  own gate.
- **Cheaper** additionally threads messaging/onboarding clarity through *every*
  milestone (per its UX/Dev Ratio Discipline), never parking it in milestone 4.
